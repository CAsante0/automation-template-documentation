Developer's Guide
==================

Overview
---------


Purpose and Goals
------------------

The goal of this workflow is to provide scientists and analysts with the ability to create fully automated data pipelines out of their existing analysis workflows. The architecture and design were done with consideration of future expansion, ease of customization, and easily integrated into the existing data infrastructure. While many tools and microservices currently exist for the development of similar workflows, most or all require payment or a subscription service. In consideration of budgetary constraints, the workflow can be run on any host that supports container-based applications with access to a github repository.

Who should read this guide?
----------------------------

Use Case
---------

The application can work with any script or module based workflow (in R or Python(in progress)) demanding data access and processing on a consistent basis (daily, monthly, etc). The current configuration files can service data platforms Copernicus Marine (in progress), OPeNDAP or ERDDAP. The modular design allows for straightforward expansion of custom modules to other data platforms that have a defined method for remote data requests. Because the full workflow is built with consideration towards robustness and resilience, it is excessive for simple workflows that rely on data import and data processing outside of the Data Import Python Library which the Controller relies on.

The purpose of this document is to outline the technical specifications, processes, and architecture of the Automation Template Application (or Automated Data Import Template).

Controller (Data Import Workflow)
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

Base Image Description
~~~~~~~~~~~~~~~~~~~~~~~

The image is based on python:3:13:3-slim for its level of security as well as its minimal design which allows for fast build times. Consideration should be given towards any further customization that relies on using any python version exceeding 3.13, in which case all operating system packages should be updated along with the base image and retested locally to ensure compatibility. Without compatible operating system dependencies the container will not build and the full containerized workflow cannot run. 

Behaviour
~~~~~~~~~~
Reads configuration file
Performs data pulls against config specified resource
Facilitates initial subsetted distribution of datasets across storage resources
Facilitates sending arguments and datasets to and from container B
Orchestrates cron schedules associated with different configuration files
Ensuring isolation of workflows by container
Logs and manifest (run metadata) are output based on the workflow specified in configuration file
